## 一、硬件约束定量分析

### 1.1 SSD1306 I2C 时序硬限制

从 SSD1306 数据手册 **Table 13-6: I2C Interface Timing Characteristics**：

| 参数 | 符号 | 最小值 | 单位 | 含义 |
|------|------|--------|------|------|
| 时钟周期 | t_cycle | **2.5** | µs | SCL 一个完整周期 |
| 起始保持时间 | t_HSTART | 0.6 | µs | Start 后 SCL↓ 前 |
| 数据建立时间 | t_SD | 100 | ns | SCL↑ 前 SDA 必须稳定 |
| 数据保持时间 (输入) | t_HD,IN | 300 | ns | SCL↑ 后 SDA 保持 |
| 起始建立时间 | t_SSTART | 0.6 | µs | 重复 Start 条件 |
| 停止建立时间 | t_SSTOP | 0.6 | µs | Stop 条件 |
| 上升/下降时间 | t_R / t_F | ≤300 | ns | SCL/SDA 边沿斜率 |
| 空闲时间 | t_IDLE | **1.3** | µs | 两次传输之间 |

**核心结论：SSD1306 I2C 时钟硬上限 = 1/2.5µs = 400 kHz**

> 当前软件 I2C（`GPIO_WriteBit`）有效 SCL ≈ 300~400kHz，**已经触及理论极限！**  
> 纯靠提高 I2C 速度来提升帧率的空间几乎为零。

### 1.2 当前一帧的全链耗时拆解

```
串口传输 (1026 bytes @ 921600 bps)
  1026 × 10 bits / 921600 ≈ 11.1 ms

I2C 刷新一帧 (8 pages × (3 cmd + 128 data) + overhead)
  8 × I2C_WriteCommand(3 bytes each): ~8 × 220 µs  ≈ 1.8 ms
  8 × I2C_WriteData(128 bytes each):  ~8 × 3.4 ms  ≈ 27.2 ms
  ─────────────────────────────────────────────────
  I2C 总计:                                       ≈ 29.0 ms

当前串行流程:
  串口 11ms → I2C 29ms → ACK → 串口 11ms → I2C 29ms → ...
  单帧 = 11 + 29 = 40 ms → 25.0 fps (理论)
  实测 = 24.7 fps  ← 完美吻合！系统已达软件 I2C 的物理极限
```

### 1.3 CH340 USB 转串口限制

| 参数 | 值 |
|------|------|
| 标称最大波特率 | **2 Mbps** |
| 稳定波特率 (Windows) | 921600 ~ 1.5M（非标准波特率需特殊驱动配置） |
| 推荐工作波特率 | **921600**（稳定可靠，无丢包风险） |

> CH340 在 2 Mbps 下需要特殊 DLL 替换和驱动配置，稳定性因操作系统而异。保持 921600 是稳妥选择。

## 三、优化方案（按安全性和收益排序）

### 方案 A：水平寻址模式 — 消除 7 次 I2C 开销 [★★★★★ 安全 + 高收益]

**当前方式**（Page Addressing Mode）：
每页需要独立的一次 I2C 传输（Start + Address + Control + Data + Stop），8 页共 8 次。

**优化方式**（Horizontal Addressing Mode）：
发送命令 `0x20, 0x00` 后，整个帧只需**一次 I2C 传输**：

```
1. WriteCommand(0x20)     // 设置寻址模式
2. WriteCommand(0x00)     // 水平寻址模式
3. WriteCommand(0x21)     // 列范围设置
4. WriteCommand(0)        // 列起始 = 0
5. WriteCommand(127)      // 列结束 = 127
6. WriteCommand(0x22)     // 页范围设置
7. WriteCommand(0)        // 页起始 = 0
8. WriteCommand(7)        // 页结束 = 7
9. WriteData(DisplayBuf, 1024)  // 一次性写入全部 1024 字节！
```

**收益估算**：
- 消除 7 次 I2C Start/Stop/Address 开销：~7 × 250µs ≈ **1.8ms 节省**
- 消除 7 × 3 = 21 个命令字节传输
- 8 次 Data 传输合并为 1 次，消除中间 Start/Stop
- 预计总节省：**3~5ms / 帧**

```
新帧时间 = 29 - 4 = 25ms (I2C) + 11ms (串口) = 36ms → 27.8 fps 理论
```

| 风险 | 零（纯 SSD1306 标准命令，与 I2C 底层无关） |
|------|------|
| 兼容性 | SSD1306 全系列支持 |
| 代码改动量 | 小（只改 OLED_Update 函数 + OLED_Init 中加寻址模式配置） |

> **推荐首先实施此方案**，因为它不与任何底层 I2C 时序交互，100% 安全。

---

### 方案 B：带 TXE 检查的流水线 ACK [★★★☆☆ 中等安全 + 高收益]

**上一次失败原因分析**：

之前的双缓冲尝试是在 ISR 中无保护地直接 `USART_SendData(0xAC)`，可能存在以下问题之一：
1. USART TX 寄存器未空时强行写入导致数据损坏
2. BSRR I2C 导致的时序错乱（与流水线无关，是 BSRR 本身的问题）
3. 主循环中 `OLED_Update()` 与 ISR 中 `USART_SendData()` 对 GPIOA 的并发访问冲突

**安全修复方案**：

在 ISR 中发送 ACK 前先检查 TXE（发送寄存器空），并在 TXE 就绪时才发送；若未就绪则跳过本次 ACK 由主循环兜底。

```c
// ISR 中（修复版）：
void USART1_IRQHandler(void)
{
    // ... 帧接收完成，校验通过后 ...
    memcpy(temp_frame, &rx_buf[2], STREAM_FRAME_SIZE);
    frame_ready = 1;

    // ★ 安全发送 ACK：检查 TXE 标志，确保寄存器空闲
    if (USART_GetFlagStatus(USART1, USART_FLAG_TXE) != RESET)
    {
        USART_SendData(USART1, STREAM_ACK);
    }
    // 否则不发送，由主循环在 OLED_Update 后兜底发送

    // 重启 DMA
    DMA_SetCurrDataCounter(DMA1_Channel5, STREAM_PKT_SIZE);
    DMA_Cmd(DMA1_Channel5, ENABLE);
}

// 主循环：
void VideoStream_Process(void)
{
    if (frame_ready)
    {
        memcpy(OLED_DisplayBuf, temp_frame, STREAM_FRAME_SIZE);
        frame_ready = 0;
        OLED_Update();

        // 兜底 ACK：ISR 可能没发成功，这里补发
        // 如果 ISR 已发，PC 忽略重复的 ACK（协议容忍）
        while (USART_GetFlagStatus(USART1, USART_FLAG_TXE) == RESET);
        USART_SendData(USART1, STREAM_ACK);
        while (USART_GetFlagStatus(USART1, USART_FLAG_TC) == RESET);
    }
}
```

**收益**：
- 大多数情况下 ISR 中 ACK 成功 → PC 立即发下一帧 → 串口传输与 I2C 刷新**并行**
- 有效帧时间 = max(I2C时间, 串口时间) ≈ **29ms** → **34.5 fps** 理论

| 风险 | 中等（并行访问 GPIOA 的竞态，但 USART TX 和 GPIO 初始化是独立的） |
|------|------|
| 回退方案 | ISR ACK 失败时主循环兜底，天然降级到原始行为 |

### 方案 C：方案 A + 方案 B 组合 [★★★★★ 推荐实施]

两个方案互补且独立，叠加效果最大：

```
新帧时间 = max(25ms, 11ms) ≈ 25ms → 40 fps 理论
实测预估: 33~38 fps
```

**对比当前 24.7fps → 提升 35%~54%**

---

### 方案 D：SSD1306 时钟分频比调整 (D5h 命令) [★★☆☆☆ 低优先级]

数据手册 10.1.16 节：`Set Display Clock Divide Ratio / Oscillator Frequency (D5h)`

当前代码使用 `0xD5, 0x80`（默认值，分频比=1，频率=8）。
该命令控制 **内部显示刷新率**，不影响 I2C 通信速度。对视频流播放无直接影响，但可微调显示质量。

---

### 方案 E：USART 波特率提升至 1Mbps (CH340 兼容) [★★★☆☆ 可选]

部分 CH340 芯片在 1 Mbps 下可稳定工作（比 2M 可靠性高很多）：

| 波特率 | 串口单帧耗时 | CH340 稳定性 |
|--------|-------------|-------------|
| 921600 | 11.1 ms | ✅ 100% 稳定 |
| 1000000 | 10.3 ms | ✅ 大部分稳定 |
| 1500000 | 6.8 ms | ⚠️ 需测试 |
| 2000000 | 5.1 ms | ⚠️ 不稳定 |

配合方案 C（流水线），1Mbps 下有效帧时间 = max(25ms, 10.3ms) ≈ 25ms，与方案 C 基本相同（I2C 仍是瓶颈）。

> **结论：对于 CH340 建议保持 921600**。

### ❌ 已排除方案

| 方案 | 排除原因 |
|------|----------|
| BSRR 内联软件 I2C | 时序不满足 SSD1306 的 t_cycle ≥ 2.5µs、t_R/t_F ≤ 300ns 要求 |
| 硬件 I2C (I2C1) | `I2C_CheckEvent()` 开销抵消硬件加速，无收益 |

## 四、推荐实施顺序

```mermaid
graph LR
    A[当前: 24.7fps] --> B["方案A: 水平寻址<br/>→ 27~28fps"]
    B --> C["方案B: 流水线ACK<br/>→ 33~38fps"]
```

### Step 1：实施方案 A（安全，立即见效）

修改 `OLED_Init()` 和 `OLED_Update()`：
- 在 `OLED_Init()` 末尾的命令序列中增加水平寻址模式配置
- 将 `OLED_Update()` 从 8 次 `WriteData` 改为 1 次

**预期：27~28 fps**（提升 ~12%）

### Step 2：实施方案 B（在 A 的基础上）

修改 `video_stream.c`：
- 增加 `temp_frame[1024]` 缓冲区
- ISR 中存帧到 temp_frame + 条件发送 ACK
- 主循环从 temp_frame 取帧 + 兜底 ACK

**预期：33~38 fps**（累计提升 ~40%）

### Step 3（可选）：测试 1Mbps 波特率

如果 Step 2 后 I2C 不再是唯一瓶颈，可尝试 `STREAM_BAUDRATE = 1000000` 缩小串口传输时间。

## 五、实施风险与回退机制

| 方案 | 最坏情况 | 回退方式 |
|------|---------|----------|
| A (水平寻址) | 屏幕不显示 | 删除 `oled.c` 中的 0x20/0x21/0x22 命令，恢复原始 `OLED_Update` 即可 |
| B (流水线) | ACK 丢失导致 PC 端超时 | `pc_streamer.py` 已有 3 次超时自动退出机制；恢复原始 `video_stream.c` 即可 |

两个方案均可**独立回退、互不依赖**。
