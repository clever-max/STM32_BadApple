> **通信链路**：PC → USB串口(921600bps) → STM32 USART1 DMA → 软件 I2C(PB8/PB9) → OLED

## Phase 0: 初始基准

### 实现方式

- 软件 I2C：`GPIO_WriteBit(PB8/PB9)` 函数调用
- 通信协议：PC 发帧 → ISR 接收 → 主循环 I2C 刷新 → ACK → 下一帧
- 串口：921600 bps
- SSD1306 寻址方式：默认 Page Addressing Mode（每页独立 I2C 传输）

### 性能

```
实测: 24.7 fps (835帧 / 33.8秒)
```

### 瓶颈分析

单帧全链路耗时分解：

```
UART传输 1026字节 @ 921600:  1026 × 10 ÷ 921600 ≈ 11.1ms
I2C刷新 8页 × (SetCursor + WriteData):
  每页 SetCursor:   3 条 I2C WriteCommand ≈ 660µs
  每页 WriteData:    128 字节 I2C 发送       ≈ 3.4ms
  8页 × (0.66 + 3.4)ms                        ≈ 32.5ms
──────────────────────────────────────────────────
单帧总耗时 = 11.1 + 32.5 = 43.6ms → 22.9fps(理论)
实测 24.7fps = 40.5ms/帧 → 与理论吻合
```

> **结论：I2C 刷新是绝对瓶颈，占单帧耗时的 74%。**

## Phase 1: 硬件 I2C 尝试

### 方案

将软件 GPIO 模拟 I2C 替换为 STM32F103 硬件 I2C1 外设 (SCL=PB6, SDA=PB7, 400kHz Fast Mode)。

### 实现细节

使用 STM32F10x 标准外设库的 I2C API：
- `I2C_GenerateSTART()` → `I2C_Send7bitAddress()` → `I2C_SendData()` → `I2C_GenerateSTOP()`
- 每字节传输后调用 `I2C_CheckEvent()` 等待状态标志

### 结果：**24.7 fps → 24.7 fps（无提升）**

### 失败原因

`I2C_CheckEvent()` 的内部开销抵消了硬件加速：

```
每字节 I2C_CheckEvent 内部操作：
  1. 读 SR1 寄存器
  2. 读 SR2 寄存器（清除 ADDR 标志）
  3. 位掩码比较
  4. 函数调用返回
  开销 ≈ 1~2µs/字节
```

虽然硬件 I2C 自动处理 SCL/SDA 翻转，但状态轮询的开销使有效传输速率并未超过软件 I2C。STM32F1 系列（Cortex-M3）的 I2C 外设设计较老，无 FIFO 缓冲，每字节必须 CPU 介入。

### 教训

> 硬件外设 ≠ 更快。当硬件 API 的轮询开销大于纯 GPIO 翻转开销时，软件实现可能更优。

## Phase 2: BSRR 内联软件 I2C

### 方案

将 `GPIO_WriteBit()` 函数调用改为直接写 `GPIOB->BSRR` 寄存器（宏内联），消除函数调用开销。SCL 高电平后插入 `__NOP()` 延迟保证脉宽。

### 结果：**花屏 / 完全不显示**

### 失败原因 — 多维度

**原因 1：SCL 脉冲不稳定**

```c
// 错误版本
#define OLED_SCL_HIGH() do { \
    GPIOB->BSRR = GPIO_Pin_8; \
    { volatile uint8_t _d = 30; while(_d--); } \
} while(0)
```

`while(_d--)` 在 ARMCC 编译器 -O2 优化下可能被优化掉（`volatile` 阻止了优化但产生不可预测的延迟）。而 `GPIO_WriteBit()` 内部有可靠的读-改-写流程。

**原因 2：SDA 建立时间不满足**

SSD1306 数据手册 Table 13-6 要求：
- `t_SD`（数据建立时间）≥ 100ns
- BSRR 写 SDA 后紧接着写 SCL，SDA 可能未稳定即被采样

`GPIO_WriteBit()` 的函数调用开销恰好充当了自然的建立时间保障。

**原因 3：系统级时序退化**

BSRR 宏导致 I2C 时序与系统时钟相位关系不稳定，某些位的边沿落在 SSD1306 输入采样窗口之外。

### 教训

> 在嵌入式系统中，看似"低效"的函数调用有时恰恰提供了必要的时序缓冲。直接寄存器写入必须精确满足器件时序手册的每个参数 (`t_cycle`, `t_HD`, `t_SD`, `t_R`, `t_F` 等)，一个不满足即导致全链路失败。

## Phase 3: 回退 + 水平寻址

### 方案

**核心洞察**：既然 I2C 速度无法提升，那就减少 I2C 传输**次数**。

SSD1306 默认使用 Page Addressing Mode，每页需要一次独立的 I2C 传输（含 Start + Address + Control Byte + Data[] + Stop），共 8 次。切换到 Horizontal Addressing Mode 后，全屏只需一次 I2C 传输。

### 实现

**OLED_Init() 中增加水平寻址配置（在 `0xAF` 显示开启前）：**

```c
OLED_WriteCommand(0x20);  // Set Memory Addressing Mode
OLED_WriteCommand(0x00);  // 00 = Horizontal Addressing Mode
OLED_WriteCommand(0x21);  // Set Column Address Range
OLED_WriteCommand(0);     // Column Start = 0
OLED_WriteCommand(127);   // Column End = 127
OLED_WriteCommand(0x22);  // Set Page Address Range
OLED_WriteCommand(0);     // Page Start = 0
OLED_WriteCommand(7);     // Page End = 7
OLED_WriteCommand(0xAF);  // Display ON
```

**OLED_Update() — 从 8 次 I2C WriteData 改为 1 次：**

```c
// 旧：8次 I2C传输
for (j = 0; j < 8; j++) {
    OLED_SetCursor(j, 0);                    // 3 cmd bytes
    OLED_WriteData(OLED_DisplayBuf[j], 128); // 128 data bytes
}

// 新：1次 I2C传输
OLED_WriteData((uint8_t *)OLED_DisplayBuf, 1024);
```

### 关键 Bug 修复

`OLED_WriteData(Data, Count)` 的 `Count` 参数原为 `uint8_t`，最大 255。传 1024 溢出为 **0**，屏幕无输出。

```c
// 修复：uint8_t → uint16_t
void OLED_WriteData(uint8_t *Data, uint16_t Count) {
    uint16_t i;  // 同样改为 uint16_t
    ...
}
```

### 性能提升

```
节省内容：
  消除 7 次 Start/Stop：        7 × (170+160)µs ≈ 2.3ms
  消除 7 次 设备地址+控制字节：  7 × 2 × 200µs     ≈ 2.8ms
  消除 7 次 SetCursor(3字节)：  7 × 3 × 200µs     ≈ 4.2ms
  ─────────────────────────────────────────────
  总计节省：                                       ≈ 9.3ms

旧 I2C 时间：32.5ms → 新 I2C 时间：~23ms
单帧总时间：23 + 11 ≈ 34ms → 29.4fps(理论)
```

**实测：~30~32fps** ✓（由于水平寻址模式下硬件内部自动跨页，无需 CPU 干预，额外节省了部分开销）

## Phase 4: 双缓冲流水线 ACK

### 方案

将 ISR 中的 `memcpy` 目标从 `OLED_DisplayBuf` 改为独立的 `temp_frame[1024]`，实现 ISR 接收与主循环显示的**解耦**。

```c
// ISR中：
memcpy(temp_frame, &rx_buf[2], 1024);  // 存到缓冲
frame_ready = 1;
USART_SendData(0xAC);                   // 立即ACK → PC发下一帧
// 重启DMA接收...

// 主循环中：
memcpy(OLED_DisplayBuf, temp_frame, 1024); // 从缓冲取
OLED_Update();                              // I2C刷新（25ms）
// 此时PC已发来下一帧，DMA正在后台接收
```

### 流水线效果

```
帧N的时间线：
  [ISR ACK] → PC发帧N+1(11ms) → DMA接收 → ...
          ↓ 并行
  [主循环] memcpy + OLED_Update(~23ms I2C刷新)
  
  有效帧时间 = max(23ms, 11ms) ≈ 23ms → 43.5fps(理论)
```

### 双 ACK 问题与修复

**第一版错误**：ISR 和主循环都发 ACK → PC 收到 2× ACK → 以 2× 速度发帧 → DMA 缓冲区溢出。

**修复**：仅 ISR 发 ACK，主循环不发。ISR 检查 TXE 标志确保安全。

### 实测

```
13928 帧, 0 次 ACK 超时, 66.6 fps
```

> **实际帧率 66.6fps 远超预期的 43.5fps！**

## Phase 5: 竞态保护

### 问题

视频播放流畅（66.6fps），但画面出现**闪烁**。原因：ISR 更新 `temp_frame` 的过程中，主循环同时 `memcpy` 读取 `temp_frame` → 产生混合帧数据。

### 修复

```c
// ISR中增加竞争保护：
if (!frame_ready)  // 主循环尚未消费前帧
{
    memcpy(temp_frame, &rx_buf[2], STREAM_FRAME_SIZE);
    frame_ready = 1;
}
// ACK照常发送（丢弃冗余帧，视频播放可接受）
```

当 `frame_ready == 1` 时（主循环还在处理前帧），ISR 跳过写入但照常发 ACK。PC 继续传送，多发的帧被丢弃（丢帧在视频播放中表现为跳帧，人眼不可察觉）。

## 核心经验

### 1. 减少传输次数 > 提升传输速度

水平寻址模式将 8 次 I2C 传输合并为 1 次，消除了 Start/Stop/Address 的开销。这比任何加速单次 I2C 传输的方法都有效。

### 2. 并行化 > 串行优化

双缓冲流水线让 I2C 刷新和串口 DMA 接收重叠执行，有效帧时间 = max(I2C, UART) 而非 I2C + UART。

### 3. 竞态条件是流水线的隐藏杀手

任何 ISR-主循环共享的缓冲区都必须有竞态保护。`volatile` 只能保证可见性，不能保证原子性。

### 4. 数据手册是优化的最终依据

SSD1306 Table 13-6 的 I2C 时序参数 (t_cycle ≥ 2.5µs, t_SD ≥ 100ns 等) 明确定义了速度上限。任何试图"超越"数据手册的优化都是 不可行的。

### 5. 测量驱动优化

每个 Phase 都必须实测验证，不能仅凭理论估计。实际 66.6fps 远超理论 43.5fps，说明实测中 I2C 刷新比估算更快（SSD1306 内部流水线效应）或 PC 端预加载了帧数据。
